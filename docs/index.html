About MergeAssist

Understanding the core concept

Use MergeAssist to analyze and merge Appian application packages. Analyze three package versions to identify required changes.

Base Version

(Package A)

Original Appian package before customizations.

Customized Version

(Package B)

Customized version of Appian package containing modifications.

New Version

(Package C)

Latest Appian package with new features, fixes, and improvements.

Benefits of Three-Way Comparison

Standard comparison tools show changes between two versions. Including the original base enables MergeAssist to distinguish between team modifications and Appian's updates to identify required merges.

Get Started

Your first merge analysis in 3 simple steps

1

Prepare Packages

Gather three Appian export packages (ZIP files) to compare:

  • Package A: Export original Appian application from baseline environment
  • Package B: Export your customized version from your development/production environment
  • Package C: Obtain the Appian's new release package
Ensure all packages are exported with the same export settings for accurate comparison.
2

Upload and Analyze

Start a new merge session and upload your packages:

  • 1. In the dashboard, click Create Merge Session.
  • 2. Enter a session name.
  • 3. Upload each package to its designated slot.
  • 4. Click Start Analysis to begin processing.
Analysis time depends on package size. Large packages (500+ objects) take a few minutes.
3

Review Results

Once analysis completes, you'll see a comprehensive breakdown:

  • View the summary dashboard with change statistics
  • Filter changes by classification, object type, or status
  • Review each change in detail with side-by-side comparisons
  • Mark changes as reviewed and add notes

Merge Workflow

Behind the scenes

MergeAssist uses a 10-step process to classify changes. Here is the workflow:

Step 1-4: Extract Packages

Extract and parse packages. Identify all Appian objects and catalog their metadata.

Step 5: Compare Delta (A → C)

Compare the base version to the new version from Appian to identify Set D (Appian changes).

Set D = Objects changed between Package A and Package C

Step 6: Compare Customization (A → B)

Compare the base version to the customized version to identify Set E (team changes).

Set E = Objects changed between Package A and Package B

Step 7: Classify Change

MergeAssist uses set theory to classify each change:

CONFLICTD ∩ E where B ≠ CBoth modified differently
COMPATIBLED \ E or (D ∩ E where B = C)Vendor-only changes or identical changes
CUSTOMISEDE \ DCustomer-only changes (additions or modifications)
DELETEDD (DEPRECATED) ∩ EVendor removed, but you modified

Step 8-9: Analyze Dependency

Analyze object dependencies to determine the optimal sequence for review.

Step 10: Generate AI Summary (Optional)

AI generates summaries of complex changes to explain what changed and why.

Understanding Classifications

What each classification means and how to handle it

COMPATIBLE Vendor Modifications

What it means

Appian-only or identical modifications.

Scenarios

  • Vendor-only change: The vendor modified or added the object, but you didn't touch it. Accept the vendor's version.
  • Vendor addition: The vendor added a new object in their latest release. Safe to accept.
  • Identical changes: Both you and the vendor made the exact same modification. Either version works.

Recommended Action

Review briefly to confirm, then mark as reviewed. These typically don't need manual intervention.

CONFLICT Requires Manual Review

What it means

Multiple disparate modifications to the same object.

Why it happens

  • You customized a component that the vendor also updated
  • Both parties fixed the same bug differently
  • Vendor refactored code you had extended

Recommended Action

Carefully review both versions. You may need to:

  • Manually merge the changes
  • Choose one version and re-apply the other's changes
  • Consult with your team or the vendor
CUSTOMISED Customer-Only Changes

What it means

Your team made changes that the vendor didn't touch. These are your customizations that are preserved during import.

Scenarios

  • Customer modification: You modified an object, but the vendor didn't change it. Keep your version.
  • Customer addition: Your team added a custom object to the application.

Common examples

  • Custom expression rules added by your team
  • Modified interfaces for enhanced UX
  • Custom integrations or connected systems
  • Bug fixes or enhancements you made

Recommended Action

These are your custom objects. Verify they are still needed and don't conflict with new vendor functionality. Usually safe to keep as-is.

DELETED Careful Review Needed

What it means

Appian removed an object that you had customized.

Why it's flagged

  • Vendor removed functionality you depend on
  • Object was replaced with a new implementation
  • Vendor deprecated legacy components

Recommended Action

Investigate why the vendor removed it. You may need to:

  • Migrate your customizations to the replacement object
  • Keep your version if the functionality is still needed
  • Remove dependencies on the deprecated object

Review Changes

Best practices for efficient change review

Use Filters Effectively

Start by filtering to show only CONFLICT items. These need the most attention. Then move to DELETED, and finally review COMPATIBLE items.

Group by Object Type

Review similar objects together. For example, review all Expression Rules first, then Interfaces. This helps you spot patterns and maintain consistency.

Follow Dependency Order

MergeAssist calculates the optimal review order based on dependencies. Objects that others depend on should be reviewed first. Use the suggested order when possible.

Add Notes

Document your decisions as you review. Notes help team members understand why certain choices were made and serve as a record for future upgrades.

Compare Side-by-Side

Use the detailed view to compare conflicting versions side-by-side.

Collaborate with Team

Involve team members who made the original modifications for complex conflicts.

FAQ

Quick answers to common questions

MergeAssist accepts standard Appian export packages in ZIP format. These are the same files you get when exporting applications from Appian Designer.

Analysis time depends on package size. Small packages (under 100 objects) typically complete in under a minute. Large packages (500+ objects) may take several minutes. You'll see a progress indicator during processing.

MergeAssist supports all major Appian object types including:

  • Interfaces
  • Expression Rules
  • Process Models
  • Record Types
  • CDTs (Complex Data Types)
  • Integrations
  • Web APIs
  • Sites
  • Groups
  • Constants
  • Connected Systems

Each merge session is tied to the packages uploaded during creation. To analyze different packages, create a new merge session. You can delete old sessions from the sessions list if they're no longer needed.

MergeAssist uses SHA-512 diff hashes to compare content. This is the same approach Appian uses internally. Two objects with the same diff hash have identical content, even if they have different version UUIDs or timestamps.

When enabled in Settings, AI generates human-readable summaries of complex changes. This helps you quickly understand what changed without reading through all the code. The feature is optional and can be disabled if not needed.

All data is stored locally in your instance's database. Uploaded packages are processed on your server and are not sent to external services (except for optional AI summaries if enabled). You can backup and restore your data from the Settings page.
